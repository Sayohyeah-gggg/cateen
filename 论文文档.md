# 基于微信小程序与 AI 大模型的食堂美食评估管理系统

---

## 一、需求分析

### 1.1 开发背景

高校及企业食堂每日服务大量用户，但传统食堂信息化程度低，用户无法提前了解菜品信息、无法便捷评价，管理员也缺乏有效的数据分析工具。本项目针对上述痛点，开发一套集美食信息展示、用户评价、社区分享、AI 智能推荐于一体的食堂美食评估管理系统。

### 1.2 竞品分析

| 对比维度 | 大众点评 | 美团外卖 | 本系统 |
|----------|----------|----------|--------|
| 目标场景 | 社会餐饮 | 外卖配送 | 食堂内部管理 |
| 用户群体 | 大众消费者 | 大众消费者 | 食堂用户 + 管理员 |
| AI 推荐 | 基于历史订单 | 基于地理位置 | 自然语言对话推荐 |
| 管理后台 | 商家端有限 | 商家端有限 | 完整 Web 管理平台 |
| 数据报表 | 无导出功能 | 有限导出 | AI 驱动 Excel/PPT 生成 |
| 社区互动 | 评论+图文 | 评论 | 帖子+评论+点赞 |
| 部署方式 | SaaS 平台 | SaaS 平台 | 私有化部署 |
| 定制化程度 | 低 | 低 | 高（可按需扩展） |

### 1.3 面向用户与主要功能

**面向用户**：食堂普通用户（通过微信小程序使用）和食堂管理员（通过 Web 后台使用）。

**主要功能**：
- 用户端：美食浏览与搜索、评分评论、收藏、榜单查看、食友分享社区、AI 智能推荐、幸运转盘随机推荐
- 管理端：美食/分类/标签/榜单/评论/用户/帖子全生命周期管理、数据统计可视化、AI 对话式数据分析、一键生成 Excel/PPT 报表

**主要性能**：接口平均响应时间 < 200ms，支持 JWT 无状态认证，AI 对话响应时间 < 3s。

## 二、概要设计

### 2.1 系统架构

系统采用前后端分离的三端架构，后端提供统一 RESTful API，分别服务微信小程序端（`/api/mini`）和 Web 管理端（`/api/admin`）。

```
┌─────────────────────────────────────────────────────────────┐
│                        客 户 端                              │
│   ┌─────────────────────┐      ┌──────────────────────────┐ │
│   │    微信小程序端       │      │      Web 管理后台         │ │
│   │  (原生小程序 WXML)   │      │  (React 18 + TypeScript) │ │
│   └──────────┬──────────┘      └────────────┬─────────────┘ │
└──────────────┼───────────────────────────────┼───────────────┘
               │ /api/mini                     │ /api/admin
┌──────────────▼───────────────────────────────▼───────────────┐
│                     Spring Boot 后端服务                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────────────┐ │
│  │  认证模块 │ │  美食模块 │ │  论坛模块 │ │  AI + 报表模块  │ │
│  │  JWT     │ │  CRUD    │ │  帖子点赞 │ │ LangChain4j+POI│ │
│  └──────────┘ └──────────┘ └──────────┘ └─────────────────┘ │
└──────────────────────────┬───────────────────────────────────┘
                           │
              ┌────────────┴────────────┐
              │  MySQL 8.0              │  Redis
              │  (主数据库，15张表)      │  (缓存/Token)
              └─────────────────────────┘
```

### 2.2 功能模块层次结构

```
食堂美食评估管理系统
├── 小程序用户端
│   ├── 美食模块（浏览、搜索、详情、收藏）
│   ├── 评论模块（发布评论、查看评论、评分）
│   ├── 榜单模块（榜单列表、榜单详情）
│   ├── 社区模块（发帖、评论、点赞）
│   ├── AI 助手模块（对话推荐）
│   └── 个人中心（收藏、历史评论）
└── Web 管理端
    ├── 内容管理（美食、分类、标签、榜单）
    ├── 用户管理（用户列表、角色、状态）
    ├── 审核管理（评论审核、帖子审核）
    ├── 数据统计（仪表盘、图表）
    └── AI 助手（数据查询、报表生成）
```

### 3.2 关键技术实现

#### 3.2.1 AI Tool Calling 机制

本系统的核心创新点之一是将 AI 大模型与数据库实时查询结合，实现自然语言驱动的数据分析。

**实现原理：**

```
用户输入自然语言
      ↓
LangChain4j 将 @Tool 方法描述注入 System Prompt
      ↓
通义千问模型判断意图，决定调用哪个工具及参数
      ↓
框架自动调用对应 Java 方法，查询 MySQL 数据库
      ↓
将查询结果返回给模型，生成自然语言回复
      ↓
返回给用户
```

**示例：** 用户输入"最近7天新增了多少用户？"，模型自动调用 `getActivityStatistics(7)`，查询数据库后返回真实数据，无需人工编写查询逻辑。

**报表生成：** 用户输入"帮我生成一份包含用户和美食的Excel"，模型调用 `generateExcel("users,foods")`，系统自动生成多 Sheet Excel 文件并返回下载链接。

#### 3.2.2 JWT 认证流程

```
1. 用户提交用户名/密码
2. 后端 BCrypt 校验密码
3. 生成 JWT Token（含用户ID、角色、过期时间）
4. 客户端存储 Token（localStorage / wx.setStorageSync）
5. 后续请求携带 Authorization: Bearer <token>
6. 拦截器解析 Token → 存入 ThreadLocal（UserContext）
7. Controller 通过 UserContext.getUserId() 获取当前用户
```

#### 3.2.3 评论审核与评分联动

用户发布评论后状态为 `pending`，管理员审核通过（`approved`）后，系统自动重新计算该美食的综合评分：

```
新评分 = SUM(已通过评论的评分) / COUNT(已通过评论)
```

评分和评价人数实时更新到 `foods` 表，小程序端展示最新数据。

### 3.3 典型使用流程（小程序端）

```
启动小程序
    ↓
微信授权登录（wx.login → 后端换取 OpenID → 生成 Token）
    ↓
首页浏览美食列表（支持分类筛选、关键词搜索）
    ↓
点击美食 → 详情页（图片、价格、评分、评论列表）

    ↓
AI 调用数据库工具查询 → 返回个性化推荐
```

#### 4.1.3 发现的问题与修正

## 五、安装及使用

### 5.1 环境要求

| 环境 | 版本要求 |
|------|----------|
| JDK | 17+ |
| MySQL | 8.0+ |
| Node.js | 18+（前端构建） |
| 微信开发者工具 | 最新稳定版 |

### 5.2 后端安装步骤

```bash
# 1. 创建数据库
mysql -u root -p -e "CREATE DATABASE cateen DEFAULT CHARSET utf8mb4;"

# 2. 导入数据库脚本
mysql -u root -p cateen < cateen.sql

# 3. 修改配置文件 application.yml
#    - 数据库连接信息
#    - Redis 连接信息
#    - AI API Key（通义千问 DashScope Key）
#    - 文件上传路径和访问地址

# 4. 构建并运行
mvn clean package -DskipTests
java -jar target/cateen-backend-1.0.0.jar
```

### 5.3 Web 管理端安装步骤

```bash
cd cateen-fronted/app-6gulfhglmz29
npm install
npm run build
# 将 dist/ 目录部署到 Nginx 或直接开发模式运行：
npm run dev
```

### 5.4 小程序端安装步骤

1. 使用微信开发者工具打开 `cateen-mini/` 目录
2. 修改 `utils/config.js` 中的后端地址（`lan` 或 `prod` 环境）
3. 填写 `project.config.json` 中的 `appid`
4. 点击"预览"或"上传"发布

### 5.5 典型使用流程

**管理员流程：**
1. 访问 Web 管理端 → 使用管理员账号登录
2. 在"美食管理"中添加菜品信息（名称、分类、价格、图片）
3. 在"榜单管理"中创建榜单并关联美食
4. 在"评论管理"中审核用户提交的评论
5. 在"AI 助手"中输入"生成本周数据报告"，一键获取 Excel/PPT

**用户流程：**
1. 打开微信小程序 → 微信授权登录
2. 首页浏览美食，点击查看详情
3. 对喜欢的美食发表评论和评分
4. 在"分享"页浏览和发布食友帖子
5. 在"AI"页输入"推荐一个辣的菜"获取个性化推荐

## 六、项目总结

### 6.1 开发过程感悟

本项目从需求调研到最终实现，经历了架构设计、技术选型、编码实现、联调测试等完整的软件开发周期。

**任务分解方面**，将系统拆分为后端 API、Web 管理端、小程序端三个相对独立的子项目，各端通过统一的 RESTful 接口协作，降低了耦合度，也便于并行开发。

**面对困难方面**，AI 功能的集成是最大的挑战。最初尝试直接调用 API 拼接 Prompt，但难以维护且无法实现动态数据查询。后来引入 LangChain4j 框架，通过 Tool Calling 机制将 AI 与数据库解耦，模型只负责理解意图，数据查询由 Java 方法完成，最终实现了稳定可靠的 AI 数据分析功能。

**水平提升方面**，本项目让我深入理解了 JWT 无状态认证、MyBatis-Plus 的 Lambda 查询、Apache POI 文件生成等技术，也对 AI 大模型的工程化落地有了实践认知。

### 6.2 后续升级方向

### 6.3 商业推广思考

本系统定位为私有化部署的食堂管理 SaaS 工具，可面向高校、企业园区、医院等有食堂管理需求的机构推广。核心竞争力在于 AI 驱动的数据分析和报表生成，能显著降低管理员的运营成本，具备一定的商业化潜力。
